# Шаблон промпта для анализа коммитов (xAI Grok / OpenAI / Ollama)

Этот промпт оптимизирован для **API-провайдеров**: xAI (Grok), OpenAI (GPT), Ollama.

> **Почему другой шаблон?**
> Модели типа Grok и GPT воспринимают длинные структурированные инструкции
> как «задачи к планированию» и начинают пересказывать их вместо анализа кода.
> Для этих моделей нужен короткий, прямой и максимально конкретный промпт.

Вставь содержимое (внутри тройных бэктиков) в **Settings → Analysis prompt template**.

Поддерживаемые переменные: `{projectName}`, `{projectPath}`, `{testsPath}`, `{gitOutput}`

> **Примечание:** `{testsPath}` — строковая подстановка пути к папке с тестами.
> Список релевантных тест-кейсов бэкенд добавляет в промпт автоматически.
> Поле `notes` тоже заполняется автоматически.

---

```
Ты — аналитик кода. Прочитай diff ниже и напиши технический разбор изменений в проекте {projectName}.

Не выполняй команды, не запрашивай разрешений — только текст анализа.

Твой ответ должен содержать разделы:
## {Автор} — {сообщение коммита}
**Файл:** `путь/к/файлу` | +N, -N
### Что изменено
### Причина / контекст
### Что проверить при тестировании   (если релевантно; тесты: {testsPath})
### Влияние на GUI   (если релевантно)

Глубина: тривиальный коммит — 3–5 пунктов; средний — основные разделы; крупный — все разделы.
Не повторяй diff в ответе. Отвечай на русском.

{gitOutput}
```

---

## Советы по настройке

- **Grok**: работает лучше с коротким промптом. Избегай маркированных списков
  в части инструкций — они воспринимаются как задачи к разбору, а не директивы.
- **GPT-4o / GPT-4o-mini**: воспринимает структуру лучше Grok, но всё равно
  предпочитает короткий прямой промпт для технического анализа.
- **Ollama (локальные модели)**: качество сильно зависит от модели. Qwen2.5 и
  Mistral справляются лучше Llama-based моделей для анализа кода.
