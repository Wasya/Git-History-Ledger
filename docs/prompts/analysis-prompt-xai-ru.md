# Шаблон промпта для анализа коммитов (xAI Grok / OpenAI / Ollama)

Этот промпт оптимизирован для **API-провайдеров**: xAI (Grok), OpenAI (GPT), Ollama.

> **Почему другой шаблон?**
> Модели типа Grok и GPT реагируют на структурированные инструкции иначе, чем Claude.
> Когда шаблон содержит заголовки разделов вперемешку с директивами,
> Grok воспринимает их как «задачи к планированию» и начинает пересказывать
> инструкции вместо анализа кода. Для этих моделей нужен короткий,
> прямой и максимально конкретный промпт без декоративных маркеров.

Вставь содержимое в **Settings → Analysis prompt template**.

Поддерживаемые переменные: `{projectName}`, `{projectPath}`, `{gitOutput}`

---

```
Ты аналитик кода. Прочитай diff ниже и напиши технический разбор изменений в проекте {projectName}.

Твой ответ должен содержать ровно три раздела с такими заголовками:
## Что изменилось
## Характер изменений
## Ключевые файлы

Пиши кратко и по существу. Не объясняй свой процесс, не пересказывай инструкции — сразу пиши разбор.

{gitOutput}
```

---

## Пояснения по разделам

| Раздел | Что писать |
|---|---|
| **Что изменилось** | Какие модули, компоненты, слои архитектуры затронуты (на основе путей файлов) |
| **Характер изменений** | Тип: баг-фикс / новая функция / рефакторинг / конфиг / зависимости |
| **Ключевые файлы** | 2–5 наиболее значимых файлов с кратким пояснением что в них изменилось |

## Советы по настройке

- **Grok**: работает лучше с коротким промптом. Если добавляешь инструкции — избегай
  маркированных списков в части инструкций, они воспринимаются как задачи к разбору.
- **GPT-4o / GPT-4o-mini**: воспринимает структуру лучше Grok, но всё равно предпочитает
  короткий прямой промпт для технического анализа.
- **Ollama (локальные модели)**: качество сильно зависит от модели. Qwen2.5 и Mistral
  справляются лучше Llama для технического анализа кода.
